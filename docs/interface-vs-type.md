# Interface vs Type

In this project, the choice between `interface` and `type` follows strict Clean Architecture principles and TypeScript compilation best practices.

## When to use `type` (Type Aliases)

We use `type` for data structures that are derived, fixed, and act as closed data contracts (such as API request payloads).

### Example in Core:

```typescript
export type CreateAppointmentInput = z.infer<typeof CreateAppointmentSchema>;
```

### Architectural Reasons:

1. **Immutability of Inputs:** A `type` acts as a static alias. Once inferred from **Zod**, it cannot be accidentally reopened or extended with magical fields by other layers.
2. **Value Mappings:** Perfect for primitive aliases, unions, and tuples generated by third-party validation engines.

---

## When to use `interface`

We use `interface` to model first-class citizens of the system, such as business Entities, database rows, or architectural port contracts.

### Example in Core:

```typescript
export interface AppointmentEntity extends CreateAppointmentInput {
  readonly appointmentId: string;
  readonly status: AppointmentStatus;
  readonly createdAt: string;
}
```

### Architectural Reasons:

1. **Clean Inheritance (`extends`):** Interfaces are natively designed for contract extension across clean architecture layers (e.g., adding infrastructure metadata to a domain input).
2. **Compiler Performance (`tsc`):** TypeScript caches interface shapes internally by name. This makes type checking drastically faster compared to type intersections (`&`), which are evaluated dynamically on every compiler pass.
3. **Readable Error Messages:** When type validation fails, `tsc` emits clear errors naming the specific `interface` and property, instead of dumping a massive, unreadable raw object structure in the terminal.
